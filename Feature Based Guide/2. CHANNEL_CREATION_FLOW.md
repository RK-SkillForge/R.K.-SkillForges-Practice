1. DETECT "USER HAS NO CHANNEL" AFTER AUTH
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      A way for the frontend to know, right after login/signup,
            whether this user already owns a channel.
Hints:
  >>> Two options: (a) add `has_channel: bool` to your existing /me
      response, or (b) a dedicated GET /channels/mine call. Fewer
      round-trips usually wins for something checked on every login.
  >>> Decide now: is creating a channel mandatory (block the app until
      done) or skippable (show once, allow "do it later")? This
      decision shapes your modal's UX — a mandatory flow shouldn't
      have a visible close button.
  >>> Don't infer "has channel" purely on the frontend from local
      state — always trust the backend's answer, since a user could
      log in on a new device/browser.
Official docs:
  FastAPI response models: https://fastapi.tiangolo.com/tutorial/response-model/
  React state after async login (useEffect + conditional render):
      https://react.dev/reference/react/useEffect

2. CHANNEL DATA MODEL (MONGODB)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      A `channels` collection, one document per channel, linked
            to the owning user.
Hints:
  >>> Store `owner_id` referencing the user's _id — don't embed the
      full user document inside the channel (keeps them independently
      updatable).
  >>> Typical fields to plan for: name, handle (unique), description,
      category, avatar_url, banner_url, created_at.
  >>> Think about 1-channel-per-user vs multi-channel-per-user now —
      it changes whether owner_id needs a unique index.
Official docs:
  MongoDB schema design patterns: https://www.mongodb.com/docs/manual/data-modeling/
  MongoDB references vs embedding: https://www.mongodb.com/docs/manual/core/data-model-design/
  Pydantic models for MongoDB docs (via FastAPI): https://fastapi.tiangolo.com/tutorial/body/

3. MODAL UI TRIGGER + COMPONENT STRUCTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      A modal component that opens automatically post-auth when
            has_channel is false.
Hints:
  >>> Keep the modal's open/close state lifted to wherever your auth
      success handler lives, not buried inside the modal itself.
  >>> Trap focus inside the modal and handle Escape/backdrop-click
      deliberately — especially important if this is a mandatory step.
  >>> If using shadcn/ui or Radix under the hood, their Dialog
      primitive already handles focus-trapping and a11y for you.
Official docs:
  React conditional rendering: https://react.dev/learn/conditional-rendering
  Radix Dialog (accessible modal primitive): https://www.radix-ui.com/primitives/docs/components/dialog
  WAI-ARIA modal dialog pattern: https://www.w3.org/WAI/ARIA/apg/patterns/dialog-modal/

4. CHANNEL FORM FIELDS + VALIDATION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      Form inside the modal — name, handle, description,
            category, avatar/banner.
Hints:
  >>> Validate handle format client-side (lowercase, alphanumeric +
      hyphen, length limits) before it ever hits the network — cheap
      to check, saves a round-trip.
  >>> Mirror every client-side rule on the backend with a Pydantic
      validator — never trust the frontend alone.
  >>> Category could be a fixed enum (dropdown) rather than free text —
      makes filtering/search much easier later.
Official docs:
  React Hook Form (common form library): https://react.hook-form.com/get-started
  Pydantic field validation: https://docs.pydantic.dev/latest/concepts/fields/
  Pydantic custom validators: https://docs.pydantic.dev/latest/concepts/validators/

5. HANDLE / SLUG UNIQUENESS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      Guarantee no two channels share a handle (this is your
            public URL slug, e.g. skillforge.com/c/your-handle).
Hints:
  >>> Enforce a unique index on `handle` in Mongo — this is your real
      guarantee, not just an app-level check (which has a race window).
  >>> Consider a live "checking availability..." lookup as the user
      types the handle, debounced, hitting a lightweight GET endpoint.
  >>> Decide how you handle the unique-index violation on submit —
      catch it and return a clean 409, don't let a raw DB error reach
      the frontend.
Official docs:
  MongoDB unique indexes: https://www.mongodb.com/docs/manual/core/index-unique/
  FastAPI custom exception handling: https://fastapi.tiangolo.com/tutorial/handling-errors/
  HTTP 409 Conflict: https://developer.mozilla.org/en-US/docs/Web/HTTP/Status/409

6. AVATAR / BANNER IMAGE UPLOAD
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      Image upload for channel avatar and banner, stored in S3.
Hints:
  >>> Don't stream the image through your FastAPI server as a proxy —
      generate a pre-signed S3 upload URL and let the browser upload
      directly to S3. Much lighter on your backend.
  >>> Validate file type and size limits before generating the
      pre-signed URL, not after the upload completes.
  >>> Store just the resulting S3 URL (or key) in the channel document,
      not the binary.
Official docs:
  S3 pre-signed URLs (boto3):
      https://boto3.amazonaws.com/v1/documentation/api/latest/reference/services/s3/client/generate_presigned_url.html
  S3 pre-signed URL upload guide:
      https://docs.aws.amazon.com/AmazonS3/latest/userguide/PresignedUrlUploadObject.html

7. POST /channels ENDPOINT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      Authenticated endpoint that creates the channel document.
Hints:
  >>> This endpoint must read the user identity from the JWT cookie
      (the auth dependency you already built), never from a field the
      client sends in the request body — otherwise anyone could create
      a channel "owned by" someone else.
  >>> Return the created channel object so the frontend can immediately
      redirect to it or close the modal with confirmation.
  >>> Consider what happens on partial failure — e.g. image uploaded
      to S3 but the DB write fails. Decide your cleanup story now.
Official docs:
  FastAPI dependency injection (reusing your auth check):
      https://fastapi.tiangolo.com/tutorial/dependencies/
  FastAPI request body + response: https://fastapi.tiangolo.com/tutorial/body/

8. WIRING IT ALL TOGETHER
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      Connect login/signup success → has_channel check → modal
            → form submit → redirect to new channel.
Hints:
  >>> Test the full loop manually as three separate users: signup flow,
      login flow (existing user without a channel), login flow
      (existing user WITH a channel — modal should NOT reappear).
  >>> If you built Google OAuth in the previous roadmap, this same
      check needs to fire after that callback too — don't duplicate
      the logic, funnel both auth paths through one shared "post-auth"
      handler.
Official docs:
  (No new doc here — this step is about revisiting steps 1–7 and your
  existing auth flow from AUTH_ROADMAP.md to make sure they connect.)

KEY TAKEAWAYS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
>>> Trust the backend, not frontend state, for "does this user have
    a channel" — it's the single source of truth.
>>> Uniqueness constraints (handle) belong in the database, not just
    app logic — the DB is your real guarantee against race conditions.
>>> Auth identity for channel ownership comes from the JWT, never from
    a client-supplied field — this is the same trust boundary
    principle from your auth roadmap, applied to a new feature.
