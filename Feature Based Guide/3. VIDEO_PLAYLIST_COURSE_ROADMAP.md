1. DATA MODEL: VIDEO, PLAYLIST, COURSE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      Three related collections in Media DB (MongoDB Atlas)
Hints:
  >>> Video needs a `source` field: "upload" | "youtube" — the two
      creation paths (sections 2 and 3) both write to the same
      collection, just with different fields populated
  >>> Playlist stores an ORDERED array of video references (ObjectIds
      or slugs), not embedded video docs — videos can belong to
      multiple playlists, so reference, don't duplicate
  >>> Course references playlists/videos it bundles, plus commerce
      fields: price, currency, is_paid, external_checkout_url
  >>> Decide creator ownership up front — every Video/Playlist/Course
      doc needs a `creator_id` so you can enforce "only the owner can
      edit this" later
Official docs:
  MongoDB referenced one-to-many modeling:
      https://www.mongodb.com/docs/manual/tutorial/model-referenced-one-to-many-relationships-between-documents/
  MongoDB schema design patterns overview:
      https://www.mongodb.com/docs/manual/data-modeling/design-patterns/

2. CREATOR UPLOADS A VIDEO (EXISTING PIPELINE, RECAP)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      Nothing new here — this is your existing architecture.
            Creator → Upload Service → Object Storage → Message Queue
            → Transcoding → Processed Storage, with a Video doc
            written to Media DB once transcoding completes.
Hints:
  >>> The Video doc created here is the same collection sections 3–5
      write into — this is why the `source` field matters, so your
      frontend can render "uploaded" vs "YouTube-embedded" players
      differently
  >>> Don't mark the video "ready" in the DB until transcoding
      finishes — the queue consumer should update status, not the
      initial upload request
Official docs (same as your architecture diagram):
  AWS S3 multipart upload (large video files):
      https://docs.aws.amazon.com/AmazonS3/latest/userguide/mpuoverview.html
  SQS or similar queue for decoupling upload from transcoding:
      https://docs.aws.amazon.com/AmazonSQS/latest/dg/welcome.html

3. FETCH A VIDEO BY YOUTUBE URL (IMPORT PATH)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      Creator pastes a YouTube URL → you fetch its metadata →
            save a Video doc with source="youtube" (no transcoding,
            no object storage — you just embed/link the original)
Hints:
  >>> Two ways to get metadata, pick based on need:
      - YouTube oEmbed (no API key, just title/thumbnail/author —
        good enough if you're only showing a preview card)
      - YouTube Data API v3 (API key required, gives duration,
        description, tags — needed if you want richer metadata)
  >>> Extract the video ID from the URL yourself (regex on the `v=`
      param or the youtu.be short form) before calling either API
  >>> Playback for a YouTube-sourced video is just the YouTube
      iframe/embed player — you are NOT pulling the actual video
      bytes into your Object Storage
  >>> Respect YouTube's Terms of Service — you're linking/embedding,
      not rehosting their content as your own
Official docs:
  YouTube oEmbed endpoint:          https://www.youtube.com/oembed
  oEmbed spec (general):            https://oembed.com/
  YouTube Data API v3 (videos.list): https://developers.google.com/youtube/v3/docs/videos/list
  YouTube IFrame Player API (embed): https://developers.google.com/youtube/iframe_api_reference
  YouTube API Terms of Service:     https://developers.google.com/youtube/terms/api-services-terms-of-service

4. CREATE A PLAYLIST + ADD VIDEOS TO IT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      POST /playlists, POST /playlists/{id}/videos
Hints:
  >>> Store video order as an array of {video_id, position} — plain
      array-of-ids works too, but an explicit position field makes
      reordering (drag-and-drop later) much simpler to update
  >>> A video can be uploaded OR youtube-sourced — the playlist
      doesn't care which, it just references Video docs by id, so
      this endpoint is identical regardless of source
  >>> Adding to a playlist should validate the video and playlist
      belong to the same creator, unless you're intentionally
      building a "curate from anywhere" feature
Official docs:
  MongoDB array update operators ($push, $pull, positional updates):
      https://www.mongodb.com/docs/manual/reference/operator/update-array/
  FastAPI path operations for nested resources:
      https://fastapi.tiangolo.com/tutorial/path-params/

5. FETCH A PLAYLIST BY YOUTUBE URL (IMPORT A WHOLE YT PLAYLIST)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      Creator pastes a YouTube PLAYLIST url → you import every
            video in it as Video docs, then create a Playlist doc
            referencing all of them
Hints:
  >>> This needs the Data API (oEmbed doesn't do playlists) —
      `playlistItems.list` gives you the video IDs in order, then
      you still call `videos.list` for each one's details
  >>> YouTube API has a daily quota — batch requests where you can
      (videos.list accepts up to 50 IDs in one call) instead of
      looping one-by-one
  >>> Long playlists (100+ videos) means pagination — the API
      returns a `nextPageToken`, don't assume one page is everything
Official docs:
  playlistItems.list:  https://developers.google.com/youtube/v3/docs/playlistItems/list
  videos.list (batch by id): https://developers.google.com/youtube/v3/docs/videos/list
  API quota costs:      https://developers.google.com/youtube/v3/determine_quota_cost

6. VIEWER/CREATOR WATCH EXPERIENCE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      GET /videos/{id}, GET /playlists/{id} for both roles
Hints:
  >>> Reuse your existing Redis metadata cache here (section from
      your architecture diagram) — playlist/video metadata is read
      far more often than it's written, ideal cache candidate
  >>> Uploaded videos stream from CDN (Processed Storage origin);
      YouTube videos render via the IFrame embed — your frontend
      player component should branch on `source`, same field from
      section 1
  >>> Creator view needs edit affordances (reorder, remove); viewer
      view doesn't — same data, different UI permissions, check this
      server-side too, not just by hiding buttons in the UI

7. PAID COURSE: CREATION + DETAILS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      POST /courses — title, description, thumbnail, price,
            currency, playlists/videos included, external_checkout_url
Hints:
  >>> `external_checkout_url` is the link to YOUR OWN payment/website
      where the actual charge happens — this is a link field, not a
      payment integration, since you said checkout lives on your site
  >>> Consider a `course_id` (or similar) as a query param you append
      to that checkout URL when redirecting, so your own site knows
      which course was purchased
  >>> Price as an integer in the smallest currency unit (paise/cents)
      avoids floating-point rounding bugs — store 49900, not 499.00
Official docs:
  Stripe's currency-handling guidance (concept applies even without
  Stripe — smallest-unit integers is the standard practice):
      https://docs.stripe.com/currencies#zero-decimal

8. PAID COURSE: "BUY" BUTTON → YOUR PAYMENT SITE → ACCESS GRANT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      Buy button redirects out to your own site; after payment,
            SkillForge needs to find out and grant the user access.
Flow:       click Buy → redirect to external_checkout_url?course_id=X
            &user_id=Y → user pays on your own site → your site tells
            SkillForge the purchase succeeded → enrollment recorded
Two ways your own site can "tell" SkillForge — pick one:
  >>> Webhook (recommended): your payment site calls a SkillForge
      endpoint (e.g. POST /webhooks/purchase-complete) server-to-
      server after payment succeeds. Verify the caller with an HMAC
      signature (shared secret), not just "any request that hits
      this URL" — otherwise anyone can fake a purchase
  >>> Redirect-back with signed token: your payment site redirects
      the browser back to SkillForge with a signed, short-lived token
      proving payment succeeded; SkillForge verifies the signature
      and expiry before granting access. Simpler to set up than a
      webhook, but relies on the user's browser completing the
      redirect (less reliable than a server-to-server call)
  >>> Either way, record an Enrollment doc (user_id, course_id,
      purchased_at) rather than just flipping a boolean — you'll want
      the history for refunds/support later
Official docs:
  Python hmac module (verify webhook signatures):
      https://docs.python.org/3/library/hmac.html
  General webhook security practices:
      https://www.svix.com/resources/webhook-best-practices/
  PyJWT (for the signed-token redirect-back approach, same library
  you're already using for auth in the other roadmap doc):
      https://pyjwt.readthedocs.io/en/stable/

9. ACCESS CONTROL FOR PAID CONTENT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      Make sure paid video URLs aren't just... public
Hints:
  >>> A plain CDN URL is guessable/shareable — anyone with the link
      can watch a paid video without buying it, unless you protect it
  >>> CloudFront signed URLs/cookies let you issue time-limited,
      per-user playback access instead of a permanently public link
  >>> Check enrollment (does an Enrollment doc exist for this
      user+course?) server-side before issuing a signed URL — never
      trust a frontend-only paywall check
Official docs:
  CloudFront signed URLs & cookies:
      https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/private-content-signed-urls.html

KEY TAKEAWAYS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
>>> One Video collection, two sources ("upload" vs "youtube") — don't
    build two parallel systems, branch on a field instead.
>>> Playlists reference videos, they don't own or duplicate them —
    the same video can live in multiple playlists/courses.
>>> "Buy" is a redirect, not a payment integration — but the RETURN
    path (webhook or signed redirect) is where the real engineering
    is. A button that redirects out is easy; trustworthily finding
    out payment succeeded is the actual feature.
>>> Never rely on a public CDN URL as your paywall — verify
    enrollment server-side before issuing any playback access.
