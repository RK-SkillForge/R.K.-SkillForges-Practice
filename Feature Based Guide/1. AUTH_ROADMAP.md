1. DOCKER COMPOSE SKELETON
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      docker-compose.yml with 3 services — frontend, backend, mongo
Hints:
  >>> Use named volumes for mongo data so you don't lose it on `down`
  >>> Bind-mount your source code for hot reload; don't COPY it in dev
  >>> `depends_on` doesn't wait for Mongo to be *ready* — only *started*.
      Look up healthchecks before you rely on startup order.
Official docs:
  Compose file reference: https://docs.docker.com/reference/compose-file/
  Compose "Getting started":  https://docs.docker.com/compose/gettingstarted/
  Mongo official image:       https://hub.docker.com/_/mongo

2. BACKEND ↔ DATABASE CONNECTION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      FastAPI app with a /health endpoint that pings MongoDB
Hints:
  >>> Use Motor (async driver), not PyMongo directly — FastAPI is async-first
  >>> Store the connection string in an env var, never hardcoded
  >>> Create the Motor client once at app startup, not per-request
Official docs:
  FastAPI first steps:      https://fastapi.tiangolo.com/tutorial/first-steps/
  Motor (async Mongo driver): https://motor.readthedocs.io/en/stable/
  FastAPI + Mongo full-app guide: https://fastapi.tiangolo.com/tutorial/sql-databases/
      (SQL example, but the startup/shutdown lifecycle pattern applies)
  FastAPI lifespan events:  https://fastapi.tiangolo.com/advanced/events/

3. FRONTEND ↔ BACKEND CONNECTION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      React API client hitting your /health endpoint
Hints:
  >>> Set `withCredentials: true` now, even before cookies exist —
      you'll forget to add it later and lose an hour to CORS errors
  >>> FastAPI CORS middleware needs explicit origins when credentials
      are allowed — "*" will NOT work with credentials
  >>> Keep the base URL in an env var (.env for Vite/CRA), not hardcoded
Official docs:
  FastAPI CORS:              https://fastapi.tiangolo.com/tutorial/cors/
  Axios docs:                https://axios-http.com/docs/intro
  MDN on credentials + CORS: https://developer.mozilla.org/en-US/docs/Web/HTTP/Guides/CORS#requests_with_credentials

4. SIGNUP + LOGIN (EMAIL/PASSWORD)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      /signup and /login endpoints, users collection in Mongo
Hints:
  >>> Hash with bcrypt via passlib — never store or log the raw password
  >>> Validate email format and password strength at the Pydantic model
      level, before the DB is touched (fail loudly, fail early)
  >>> Enforce a unique index on email in Mongo — don't rely on app
      logic alone to prevent duplicate accounts
Official docs:
  FastAPI security tutorial:  https://fastapi.tiangolo.com/tutorial/security/
  Passlib (bcrypt hashing):   https://passlib.readthedocs.io/en/stable/
  Pydantic validation:        https://docs.pydantic.dev/latest/concepts/validators/
  MongoDB unique indexes:     https://www.mongodb.com/docs/manual/core/index-unique/

5. JWT SESSIONS VIA SECURE HTTPONLY COOKIES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      Access + refresh token pair, set as httpOnly cookies
Hints:
  >>> Two tokens, two lifetimes: short-lived access, longer refresh
  >>> Never put the refresh token where JS can read it — httpOnly only
  >>> Scope the refresh cookie's `path` to just your /refresh route
  >>> Read up on SameSite=Lax vs Strict BEFORE picking one — Strict
      will break your Google OAuth redirect in step 8
Official docs:
  PyJWT:                      https://pyjwt.readthedocs.io/en/stable/
  FastAPI OAuth2/JWT tutorial: https://fastapi.tiangolo.com/tutorial/security/oauth2-jwt/
  MDN Set-Cookie (httpOnly/Secure/SameSite):
      https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Set-Cookie
  OWASP session management cheat sheet:
      https://cheatsheetseries.owasp.org/cheatsheets/Session_Management_Cheat_Sheet.html

6. FORGOT / RESET PASSWORD
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      /forgot-password and /reset-password endpoints + email send
Hints:
  >>> Store a HASH of the reset token in Mongo, not the raw token —
      same principle as passwords
  >>> Give the token a short expiry (15-30 min) and mark it single-use
  >>> Respond the same way whether the email exists or not — don't
      leak which emails are registered
Official docs:
  Python secrets module (token generation): https://docs.python.org/3/library/secrets.html
  AWS SES (transactional email):  https://docs.aws.amazon.com/ses/latest/dg/Welcome.html
  FastAPI-Mail (alternative, SMTP-based): https://sabuhish.github.io/fastapi-mail/

7. RATE LIMITING
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      Limits on /login, /signup, /forgot-password
Hints:
  >>> Start app-level (fast to add), plan to move to gateway-level later
  >>> Limit by IP AND by account/email — IP-only is bypassed by rotating
      IPs, account-only is bypassed by spraying across many accounts
  >>> Decide what a rate-limited response looks like (429 + Retry-After
      header) before you build the frontend's error handling for it
Official docs:
  slowapi (FastAPI rate limiting): https://github.com/laurentS/slowapi
  MDN 429 status code:  https://developer.mozilla.org/en-US/docs/Web/HTTP/Status/429
  AWS API Gateway throttling (for later, gateway-level):
      https://docs.aws.amazon.com/apigateway/latest/developerguide/api-gateway-request-throttling.html

8. GOOGLE OAUTH LOGIN
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      /auth/google and /auth/google/callback endpoints
Hints:
  >>> Register your app in Google Cloud Console first — you need a
      client ID/secret and an authorized redirect URI before any code
  >>> On successful callback: find-or-create the user by email, then
      issue the SAME cookie pair your password login issues — one
      unified session system, not two separate ones
  >>> Test the redirect URI exactly matches what's registered — this
      is the #1 thing that breaks OAuth setups
Official docs:
  Google OAuth2 for web server apps:
      https://developers.google.com/identity/protocols/oauth2/web-server
  Google Cloud Console (create credentials): https://console.cloud.google.com/apis/credentials
  Authlib + FastAPI integration:  https://docs.authlib.org/en/latest/client/fastapi.html

9. CI/CD (GITHUB ACTIONS)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      Workflow: lint + test → build Docker image → push to ECR
Hints:
  >>> Start with a workflow that ONLY runs on pull requests (lint+test),
      add the build/push step once that's green
  >>> Use GitHub OIDC to authenticate to AWS instead of long-lived
      access keys stored as secrets — more secure, no key rotation
  >>> Cache Docker layers between runs or your pipeline will be slow
Official docs:
  GitHub Actions docs:        https://docs.github.com/en/actions
  Actions + AWS OIDC (no static keys):
      https://docs.github.com/en/actions/deployment/security-hardening-your-deployments/configuring-openid-connect-in-amazon-web-services
  Docker build-push-action:   https://github.com/docker/build-push-action

10. AWS DEPLOYMENT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Build:      ECR repo → ECS Fargate service → Application Load Balancer
Hints:
  >>> Fargate = no EC2 servers to patch/manage — worth the slightly
      higher per-hour cost for a solo project
  >>> Put secrets (JWT secret, Google client secret, Mongo URI) in
      AWS Secrets Manager or SSM Parameter Store — never in the image
  >>> Your ALB needs a health check path — reuse the /health endpoint
      from step 2
  >>> MongoDB Atlas, not self-hosted Mongo on EC2 — you're already
      using Atlas per your SkillForge architecture, stay consistent
Official docs:
  ECS Fargate overview:  https://docs.aws.amazon.com/AmazonECS/latest/developerguide/AWS_Fargate.html
  ECR user guide:        https://docs.aws.amazon.com/AmazonECR/latest/userguide/what-is-ecr.html
  Application Load Balancer: https://docs.aws.amazon.com/elasticloadbalancing/latest/application/introduction.html
  AWS Secrets Manager:   https://docs.aws.amazon.com/secretsmanager/latest/userguide/intro.html
  MongoDB Atlas docs:    https://www.mongodb.com/docs/atlas/

KEY TAKEAWAYS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
>>> Order matters: infra before features, features before deployment.
    Skipping ahead (e.g. OAuth before cookies work) creates rework.
>>> Every "hint" above tells you what to research, not what to type —
    read the linked doc section before writing the code yourself.
>>> Security defaults (bcrypt, httpOnly, hashed reset tokens, secrets
    manager) are non-negotiable even in an MVP — they're cheap to do
    right the first time, expensive to retrofit later.
